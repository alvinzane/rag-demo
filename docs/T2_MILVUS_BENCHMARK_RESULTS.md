# W4-T2 Milvus 1M 向量压测结果解读

本文记录一次 Milvus Standalone + IVF_FLAT 的向量检索压测结果，并解释相关基础概念，方便新手理解每个命令和指标的含义。

## 测试环境与目标

本次测试目标是验证 W4-T2 的 1M 向量压测链路：

- 生成确定性测试数据和查询集。
- 向 Milvus 导入向量数据。
- 构建 IVF_FLAT 索引。
- 调整 `nprobe` 参数，观察查询延迟和吞吐变化。

测试使用的集合为：

```text
rag_demo_t2_vectors
```

## 基本概念

### 向量和维度

向量是由一组数字组成的数组，可以表示文本、图片或其他对象的语义特征。例如 768 维向量表示每条数据有 768 个浮点数。

维度越高，每条向量包含的信息越多，但计算和存储成本也越高。本次完整测试使用的是：

```text
1000000 条向量，每条 768 维
```

### 向量检索

向量检索是指给定一个查询向量，从库中找出最相似的若干条向量。本次测试默认返回：

```text
top_k = 10
```

也就是每次查询返回最相似的 10 条结果。

### COSINE 相似度

本次索引使用：

```text
metric_type = COSINE
```

COSINE 表示用余弦相似度衡量两个向量方向是否接近。它常用于文本 embedding，因为很多语义相似度任务更关心方向，而不是向量长度。

### IVF_FLAT

`IVF_FLAT` 是一种常见的近似最近邻索引。

它会先把全量向量划分成多个聚类桶。查询时不扫描所有向量，而是先找到最可能相关的若干个桶，再在这些桶里做精确计算。

优点：

- 查询速度通常比全量扫描快。
- 参数简单，适合教学和基准测试。

代价：

- 它是近似检索，可能漏掉真正最相似的结果。
- 需要通过参数平衡速度和召回率。

### nlist

`nlist` 是 IVF 索引中的桶数量。

本次完整测试使用：

```text
nlist = 1024
```

对于 100 万条向量，平均每个桶大约有：

```text
1000000 / 1024 ≈ 977 条向量
```

`nlist` 越大，桶越多，每个桶平均越小。但桶太多也可能增加建索引成本，并且需要合适的 `nprobe` 配合。

### nprobe

`nprobe` 是查询时扫描多少个桶。

例如：

- `nprobe=1`：只查 1 个最相关的桶，速度快，但可能漏结果。
- `nprobe=8`：查 8 个桶，候选更多，召回率通常更好，但更慢。
- `nprobe=64`：查 64 个桶，延迟更高，但通常更接近精确搜索。

在 `nlist=1024`、100 万条向量的情况下，粗略候选规模为：

| nprobe | 约扫描桶数 | 约候选向量数 |
| --- | ---: | ---: |
| 1 | 1 | 977 |
| 4 | 4 | 3908 |
| 8 | 8 | 7816 |
| 16 | 16 | 15632 |
| 32 | 32 | 31264 |
| 64 | 64 | 62528 |

这只是平均估算，实际每个桶的大小可能不同。

### QPS 和延迟指标

本次压测输出了以下性能指标：

| 指标 | 含义 |
| --- | --- |
| QPS | 每秒完成的查询次数，越高越好。 |
| avg_ms | 平均查询耗时，越低越好。 |
| p50_ms | 50% 查询不超过该耗时，也叫中位数延迟。 |
| p95_ms | 95% 查询不超过该耗时，用来观察大多数请求的尾部延迟。 |
| p99_ms | 99% 查询不超过该耗时，用来观察更极端的尾部延迟。 |
| min_ms | 最快的一次查询耗时。 |
| max_ms | 最慢的一次查询耗时。 |
| memory_rss_mb | 进程常驻内存，单位 MB。 |

通常不能只看平均值。在线服务更关心 `p95` 和 `p99`，因为用户容易感受到慢请求。

## 测试步骤和结果

### 1. 生成测试数据描述

命令：

```bash
uv run rag-demo t2 generate --output .rag/t2 --count 1000000 --dim 768
```

结果：

```text
Count: 1000000
Dim: 768
Queries: 1000
Materialized: False
Output: .rag/t2
```

解读：

- 准备了 100 万条、768 维的测试数据配置。
- 准备了 1000 条查询向量。
- `Materialized: False` 表示没有把 100 万条全量向量全部落盘成静态文件。
- 后续导入时会按相同 seed 流式生成数据，避免一次性占用大量磁盘或内存。

### 2. 小规模冒烟测试

命令：

```bash
uv run rag-demo t2 load --count 10000 --dim 128 --reset
uv run rag-demo t2 index --index-type IVF_FLAT --nlist 128
uv run rag-demo t2 bench --index-type IVF_FLAT --runs 50 --dim 128 --nprobe 8
```

结果：

```text
Rows inserted: 10000
Index build elapsed_sec: 1.7492914898321033
QPS: 400.53
avg_ms: 2.49
p50_ms: 2.29
p95_ms: 2.91
p99_ms: 6.25
memory_rss_mb: 171.00
```

解读：

这一步主要用于确认链路可用，包括连接 Milvus、建集合、导入数据、建索引和查询。

需要注意：这个测试只有 1 万条、128 维，数据量和维度都比完整测试小很多。因此它不能直接代表 100 万条、768 维时的性能。

### 3. 完整数据导入

命令：

```bash
uv run rag-demo t2 load --count 1000000 --dim 768 --batch-size 5000 --reset
```

结果：

```text
Rows inserted: 1000000
```

解读：

Milvus 集合中成功写入 100 万条 768 维向量。

`--batch-size 5000` 表示每批导入 5000 条，避免一次性构造 100 万条向量导致客户端内存压力过大。

### 4. 构建 IVF_FLAT 索引

命令：

```bash
uv run rag-demo t2 index --index-type IVF_FLAT --nlist 1024 \
  --report .rag/t2/ivf_index.json
```

结果：

```text
collection: rag_demo_t2_vectors
index_type: IVF_FLAT
metric_type: COSINE
rows: 1000000
elapsed_sec: 143.61798938410357
params: {'nlist': 1024}
```

解读：

100 万条、768 维向量构建 IVF_FLAT 索引用时约 144 秒。

`nlist=1024` 表示索引将数据划分为 1024 个聚类桶。按照平均值估算，每个桶约 977 条向量。

### 5. nprobe 参数扫描

命令：

```bash
uv run rag-demo t2 sweep --index-type IVF_FLAT --values 1,4,8,16,32,64 \
  --runs 1000 --dim 768 --report .rag/t2/ivf_sweep.json
```

结果：

| nprobe | QPS | P50 ms | P95 ms | P99 ms |
| ---: | ---: | ---: | ---: | ---: |
| 1 | 141.26 | 6.91 | 8.34 | 11.28 |
| 4 | 139.20 | 7.09 | 8.37 | 10.27 |
| 8 | 117.18 | 7.69 | 13.04 | 17.82 |
| 16 | 103.58 | 9.27 | 11.47 | 17.19 |
| 32 | 85.22 | 11.51 | 13.54 | 17.73 |
| 64 | 64.96 | 15.19 | 17.16 | 21.11 |

## 结果分析

### nprobe 越大，查询越慢

从结果可以看到：

- `nprobe=1` 时 QPS 为 141.26，P50 为 6.91 ms。
- `nprobe=64` 时 QPS 降到 64.96，P50 增加到 15.19 ms。

这是符合预期的。`nprobe` 越大，查询扫描的桶越多，候选向量越多，计算成本越高。

### 性能下降不是线性的

从 `nprobe=1` 到 `nprobe=64`，理论扫描桶数扩大了 64 倍。但 P50 只从 6.91 ms 增加到 15.19 ms。

这说明查询耗时中存在固定开销，也可能有 Milvus 内部并行化、缓存或批处理优化。因此实际延迟不会简单地按候选数量线性增长。

### 当前性能甜点可能是 nprobe=4

从纯性能角度看，`nprobe=4` 是这次测试中比较划算的点：

```text
nprobe=4
QPS: 139.20
P50: 7.09 ms
P95: 8.37 ms
P99: 10.27 ms
```

它的 QPS 和 `nprobe=1` 几乎相同，P50 只略高，P99 还略低。

不过这只是性能结论，不是最终参数推荐。因为本次结果没有包含召回率。

### 不能只用性能决定检索参数

IVF_FLAT 是近似检索。`nprobe` 越小，速度通常越快，但可能漏掉真正最相似的结果。`nprobe` 越大，召回率通常越高，但延迟和资源消耗也更高。

因此最终参数选择应该同时看：

- QPS
- P95 / P99 延迟
- Recall@K
- 业务是否能接受漏召回

本次测试没有输出 Recall@10，所以不能判断 `nprobe=4` 的检索质量是否足够。

## 小规模测试和完整测试不能直接对比

小规模测试：

```text
10000 条，128 维
```

完整测试：

```text
1000000 条，768 维
```

两者的数据规模相差 100 倍，维度相差 6 倍。因此小规模测试的 `400.53 QPS` 不能直接和完整测试的 `141.26 QPS` 对比。

小规模测试的作用是冒烟验证，完整测试才更接近本任务目标。

## 结论

本次测试证明：

- W4-T2 的 1M 向量导入、索引构建和查询压测链路已经可用。
- 100 万条、768 维向量构建 IVF_FLAT 索引用时约 144 秒。
- 在 `nlist=1024` 下，`nprobe=1` 到 `64` 的 P50 延迟范围约为 6.91 ms 到 15.19 ms。
- `nprobe` 增大会降低 QPS，并提高延迟，这是 IVF_FLAT 的正常现象。
- 从纯性能看，`nprobe=4` 是当前结果里较好的候选点。
- 由于缺少召回率数据，不能只根据这组结果确定最终生产参数。
