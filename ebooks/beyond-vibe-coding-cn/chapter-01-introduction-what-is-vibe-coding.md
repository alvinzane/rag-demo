# 第1章 简介：什么是Vibe编程？

AI正在重塑我们构建软件的方式，引入了从自由形式提示到结构化辅助的新编程范式。想象一下，只需*描述*你想要软件做什么——就像与队友交谈一样——AI就能将这些想法转化为代码。这就是*vibe编程*的精髓，一种以提示为先、探索性的方法，你用自然语言描述你想要的内容，让大型语言模型（LLM）填补空白。这个术语最近由AI先驱Andrej Karpathy创造，用来描述这种新的编程方式，开发者在这种方式中"完全沉浸于AI辅助的氛围"。

在本书中，我将深入探讨vibe编程对专业开发者的意义，以及它与我所称的*AI辅助工程*相比和互补的关系——后者是一种更正式的增强型编程流程。我将探讨开发者的角色在这个AI优先的时代如何演变，哪些工具和工作流程可以最大化你的效率，以及如何应对让AI在你的代码库中自由发挥所带来的独特挑战。我还会研究vibe编程在哪些方面表现出色，在哪些方面遇到困难，以及如何平衡AI生成的速度与人类监督的智慧。最后，你应该能够清楚地了解如何在自己的编程实践中"利用这些氛围"——负责任且有效地——在AI时代成为一个不仅更快、更有创造力和影响力的软件产品工程师。

在本章中，我们探讨开发者的角色如何从为机器编写详细指令转变为通过表达意图与AI协作。我们将看到为什么编程中的这种"氛围转变"如此重要，它在高层次上如何运作，以及它带来的机遇和挑战。

# AI编程谱系：从Vibe编程到AI辅助工程

在过去的一年中，我观察到开发者——特别是中级和高级Web开发者——在工作流程中采用AI的方式出现了有趣的分化。谱系的一端是vibe编程。另一端是我所称的*AI辅助工程*：一种在有明确约束下将AI编织到软件开发每个阶段（从设计到测试）的规范方法。两种方法都利用了强大的AI，但它们的目标、受众和期望明显不同。在本书中，我将探讨这两个极端以及它们对现代Web开发的意义。

## Vibe编程方法：通过对话编码

在vibe编程中，你利用强大的LLM作为编程伙伴，让它们处理代码生成的繁重工作，这样你就可以专注于更高级的目标。正如《商业内幕》的一篇总结所说，vibe编程"意味着使用AI工具...来完成编码中的繁重工作，以快速构建软件"。正如英伟达CEO黄仁勋所说，感谢AI，"最热门的新编程语言"是英语，而不是Java或Python。你不再手动输入每个函数和错误修复，而是用自然语言与AI交互——勾勒功能、审查建议、根据AI的输出进行迭代。

这种方法代表了从传统编程到AI辅助开发的巨大转变。传统编码需要仔细规划、语法精确，并且常常需要艰苦的调试。Vibe编程颠覆了这个剧本："这真的不是编程——我只是看看东西，说说东西，运行东西，复制粘贴东西，而且大部分时间都有效，"Karpathy对《商业内幕》开玩笑说，强调了AI如何能以最少的手工工作将高级指令转化为工作代码。

开发者从为计算机编写详细指令转变为在AI的帮助下*编排结果*。例如，[Karpathy描述](https://oreil.ly/Ki6iJ)通过不断接受AI的建议来构建Web应用："我总是'全部接受'，我不再阅读差异了。...当我收到错误消息时，我只是复制粘贴它们。...有时LLM无法修复错误，所以我只是绕过它或要求随机更改，直到错误消失。"代码"增长"超出他通常会自己编写的范围，但项目通过迭代式提示和修复快速组合在一起。本质上，vibe编程将编码视为与你的AI配对编程员的交互式对话，而不是独自在语法和堆栈跟踪中挣扎。目标是速度和探索——以最小的摩擦获得工作解决方案。

几个趋势的融合使vibe编程成为可能。首先，现代AI编码助手（如OpenAI的Codex、ChatGPT、Anthropic的Claude等）在生成和纠正代码方面变得异常出色。在同一篇文章中，Karpathy指出这是"可能的，因为LLM...变得太好了"——它们吸收了大量的GitHub代码，可以为许多任务产生合理的解决方案。

其次，新的开发者工具已经出现，将这些模型无缝集成到编码工作流程中（稍后会详细介绍这些工具）。最后，开发者社区的心态正在发展，越来越信任AI来完成越来越大的工作块。这不再仅仅是增强版的自动完成；它是将整个函数或文件交给AI。实际上，vibe编程通常感觉像是拥有无限数量的热切初级开发者来实现你要求的任何东西——除了他们以云计算的速度工作。

vibe编程最令人瞠目结舌的承诺之一是生产力提升。早期采用者报告说，他们能够比以前快10到100倍地创建软件功能或原型。例如，[Codeium Windsurf工程师John Hoestje思考](https://oreil.ly/_nfZn)，"为什么要成为一个10倍工程师，而你可以成为一个100倍工程师？"这表明，使用合适的AI驱动IDE，非凡的生产力是可以实现的。像Windsurf这样的AI增强IDE"可以显著加速开发时间，让你实现100倍的生产力"。虽然100倍可能是一个极端情况，但即使是更保守的研究也发现了巨大的收益。

开发者可以在几秒钟内生成样板代码，在眨眼之间修复错误，甚至让AI编写测试或文档，将过去需要几天的工作流程压缩到几小时内。不再受打字速度或记忆的限制，装备AI的单个开发者通常可以在一个周末内原型化一个全栈应用——这在过去可能需要一个小团队几周时间才能完成。这不仅仅是炒作；正如我在2025年1月为《实用工程师》撰写的[博客文章](https://oreil.ly/khEfs)中指出的，调查显示*75%的开发者*已经将某种形式的AI集成到他们的工作流程中，许多公司报告开发速度提高了两位数或三位数的百分比。总之，AI配对编程员正在将神话般的"10倍工程师"转变为非常真实（且可达到的）100倍工程师现象。

要理解这是多么革命性，考虑一个具体的例子。一个开发者想要构建一个简单的Web应用，计算播客脚本中的单词数并估计阅读时间。他们没有从头开始，而是打开一个AI驱动的编码环境，*告诉*AI他们的想法。在几分钟内，AI产生了一个工作原型。然后开发者说，"让统计计数器使用鲜艳的颜色并添加PDF导出"，AI相应地更新代码。结果是一个功能工具，一键部署——所有这些都在10分钟内完成。这个真实世界的场景（由使用Replit AI的创建者[报告](https://oreil.ly/guqFZ)）展示了vibe编程如何实现由高级请求驱动的极快速迭代开发。类似地，非工程师也加入了进来：同一篇文章描述了一位被解雇的营销人员，没有编程背景，使用AI编码助手构建了100个简单的Web工具，这些工具集体登上了Product Hunt的榜首。当创建软件的障碍降到这么低时，我们不仅仅是在提高经验丰富的开发者的生产力——我们从根本上扩展了谁可以首先开发软件。

然而，vibe编程带有严重的警告。因为你将太多工作交给AI，你最终可能会得到在理想路径下"工作"的代码，但隐藏了错误或糟糕设计决策的地雷场。没有坚实的计划或约束，LLM可能生成缺乏适当错误处理、安全检查或可扩展性的解决方案。事实上，AI生成的代码有时可能建立在沙子上：它看起来很坚固，但隐藏的问题只在现实条件下才会显现。我见过开发者以创纪录的时间通过vibe方式完成了完整功能，后来却发现代码效率低下且难以维护。这种"纸牌屋"代码在压力下会崩溃。

例如，想象一下要求AI"快速制作一个用户登录系统"。AI可能快速生成一个工作认证流程，但也许它使用了简化的加密方法或已知的易受攻击库。如果你在没有深入检查的情况下部署它，你是在相信一切都是健全的。经验丰富的工程师知道这是有风险的：在生产环境中运行的代码必须被理解和信任。正如[一位专家](https://oreil.ly/ppXCf)所说，"通过vibe编程进入生产代码库显然是有风险的。我们作为软件工程师所做的大部分工作涉及改进现有系统，其中底层代码的质量和可理解性至关重要。"在极端情况下，vibe编程可以绕过这些质量门控。

另一个挑战是vibe编程倾向于淡化前期规划。传统软件工程重视为清晰和约束而设计——思考数据模型、选择适当的模式、至少写出最小的规范。Vibe编程颠覆了这一点：它从*没有脚手架*开始，直接通过提示进入实现。这可能导致曲折的开发过程。你可能通过提示将自己逼入角落——比如AI选择了你不想要的状态管理方法或库，现在你必须要么引导它回来，要么接受它。没有初始蓝图，最终架构可能是随意的。这对于快速的概念验证来说很好，但在一致性很重要的大型代码库中是有问题的。

Vibe编程本质上并不"坏"。事实上，它的出现是编程持续民主化的一部分。它降低了创建软件的门槛，就像早期的低代码平台或脚本语言一样。有动力的非工程师如果有明确的想法，可能能够仅通过vibe构建一个简单的应用。对于经验丰富的开发者，vibe编程可以是一个强大的头脑风暴工具——就像伪代码，但有即时的、可运行的结果。关键是认识到其限制。没有纪律的速度会导致脆弱的软件，所以vibe编程需要警惕的人在循环中。我经常提醒开发者（和我自己）"vibe编程不是低质量工作的借口"。它应该是解决方案的*开始*，而不是结束。

## AI辅助工程方法：与AI伙伴的结构化方法

在我们谱系的另一端是*AI辅助工程*——一种更有结构、更有条理的软件构建方式，AI作为每一步的副驾驶。在这里，开发者仍然很大程度上处于驾驶位置。AI辅助工程包括在整个传统软件开发生命周期（SDLC）中使用AI，如AI驱动的自动完成、聊天、代码迁移、错误检测、测试生成，以及细粒度（函数、模块、组件）和完整代码生成。

![图1-2. 计划优先的AI辅助工程工作流程：开发者创建规范，向AI系统提供有针对性的提示，审查生成的代码片段，并将批准的解决方案集成到他们的项目中。](images/chapter-01-figure-02-ai-assisted-engineering-workflow.png)

你从一个计划开始（即使是轻量级的），概述你需要构建的内容，并预先定义约束和验收标准。然后你有针对性地将AI工具纳入其中，以加速或增强该计划的部分内容。与提示优先的vibe编程相比，我们可以称这是"计划优先"的AI支持开发。这可能像迷你产品需求文档（功能的简短PRD）一样正式，或者像任务清单一样简单。关键区别在于你在让AI自由发挥之前将工作建立在*明确的意图和约束*基础上。

考虑一个被要求创建新的交互式仪表板组件的React开发者。在AI辅助工程方法中，他们可能首先写下组件的职责和API：

> 仪表板组件显示分析卡片列表，支持按日期范围过滤，并有刷新和导出按钮。它应该从我们的API获取数据（有适当的错误处理），并且必须遵循我们的设计系统进行样式设计。

这个大纲本质上是一个规范。开发者甚至可能快速勾画数据模型或确定要重用的现有实用函数。只有在那时他们才引入AI：例如，使用AI启用的IDE或编码助手基于该描述生成组件的骨架。AI可能提供React组件的起始实现，为数据获取和存根事件处理程序留出占位符。因为开发者提供了明确的指导，AI的输出更有可能与项目需求一致（例如使用正确的设计系统类或调用正确的API端点）。代码不是惊喜；它是格式良好的请求的产物。
