# 第2章 提示的艺术：与AI有效沟通

在vibe编程中，提示就是新的源代码。

你向AI传达意图的方式直接影响它生成的代码质量。编写一个好的提示既是艺术也是科学，通常被称为*提示工程*。本章将为你提供充分利用AI编码助手的技巧。我们将从一些关于提示为何重要的基础知识开始，然后深入探讨从简单到高级的提示技术工具箱。通过学习如何制作有效的提示以及如何迭代地完善它们，你将能够更高效、更准确地进行AI协作。

# 提示工程基础

如果vibe编程是你和AI模型之间的对话，*提示工程*就是用AI的语言说话以获得最佳结果的技能。精心制作的提示可以是无关或错误代码建议与完美解决方案之间的区别。掌握提示工程意味着理解如何有效指导AI，如何提供上下文，以及当第一个答案不太正确时如何与AI迭代。

当你用AI编程时，你本质上是通过使用自然语言*通过*AI进行编程。你提供的提示就像一种高级编程语言，AI解释器然后将其转换为实际代码。就像编译器的输出只与提供给它的源代码一样好，AI的输出只与提示一样好。

为什么提示如此重要？LLM尽管复杂，但不是读心者。它们只响应给出的输入。模糊或措辞不当的提示可能导致无关或不正确的代码，而清晰具体的提示可以在第一次尝试时就产生准确的解决方案。在传统编程中，你花时间思考算法和编写代码；在vibe编程中，你花时间思考如何向AI传达你的需求。这是"编写代码"含义的转变：你可能写一段文字而不是一个函数，但你仍然需要精确和逻辑。

将编写提示视为为一个非常字面和迂腐的初级开发者编写文档或用户故事，这个开发者会完全（且只）按照文档说的做，并且拥有大量知识但除了他们见过的模式外没有常识。如果你的指令（提示）留有解释空间，AI可能会以你无意的方式填补空白。因此，学习与AI*沟通*和学习编程语言的语法一样重要。

提示至关重要的另一个原因是可重现性和面向未来。如果你发现一个提示能够可靠地为某种模式或任务生成好的代码，那个提示就成为有价值的知识（几乎像一个片段或模板）。你可能会保存它或在类似情况下重用它。在团队中，开发者可能会相互分享有效的提示模式，类似于他们分享编码最佳实践。

最后，随着模型变得更好和更集成，它们可能允许更复杂的交互。擅长提示将让你能够快速利用新能力。例如，一些高级系统允许你附加大量指令或提供整个参考文档作为模型的上下文。知道如何构建那个输入是利用这种力量的关键。

因此，将提示写作视为一项新的基本技能。在许多方面，提示*就是*编程。主要区别在于你用一种语言（如英语）写作，然后AI将其转换为代码。但你仍然需要在描述中清晰、逻辑和预见边缘情况。

# 具体性和清晰度：编写能交付的提示

提示的黄金法则之一（我将在第3章更全面地阐述）是对你想要的内容要具体和清晰。与人类协作者不同，AI不会真正理解你提供的词语之外的目标。一个常见错误是给AI一个非常高级的提示，如"制作一个网站"，并期望奇迹。AI在具体细节下工作得更好。

始终假设它除了你提供的内容外对你的项目一无所知。包括相关细节，如编程语言、框架和库，以及相关的函数或片段。如果有错误，提供确切的错误消息并描述代码应该做什么。任何模糊或解释空间都可能导致意外的输出。

例如，不要说"写一个排序函数"，你可以说：

> 编写一个Python函数sort_by_lastname(customers)，它接受客户记录列表（每个都有first_name和last_name字段），并返回按last_name字母排序的列表。包含简短的文档字符串，并通过将缺失的姓氏视为空字符串来处理这种情况。

这个提示为语言（Python）、函数名称和目的、输入结构、排序键、额外要求（文档字符串）和边缘情况设定了清晰的期望。它可能产生你确切需要或非常接近的东西。本质上，像规范编写者一样思考：你越精确地指定任务，AI需要做的猜测工作越少，你需要的修订越少。

## 具体性的策略包括：

**提及语言或环境**
如果你想要JavaScript解决方案，要说明："编写一个JavaScript函数..."而不是"编写一个函数..."如果你想要特定框架或版本，包括那个（"使用React Hooks..."或"在Python 3中..."）。

**定义输出的范围**
你只想要一个函数？完整的文件或模块？包括测试？例如，"只提供函数实现"和"提供完整的可运行脚本"可能产生不同的响应。

**包括要求和约束**
在登录示例中，我们指定了密码长度和尝试限制。考虑边缘情况或约束，并将它们放入提示中。如果你需要代码针对性能优化或使用特定算法，要说明："使用O(n)时间和O(1)空间"或"使用二分搜索方法"。

**避免模糊引用**
不要使用没有明确先行词的*它*等词。不要说"处理它并返回结果"，而要说"处理数组并返回结果数组"。

**命名你想要的输出格式**
如果你希望AI只输出代码或带注释的代码或解释，你可以指示："只提供代码，不要解释"或"提供代码和每个步骤的简短注释"。

清晰的提示为AI成功奠定基础。如果你发现AI的答案经常需要大量修正，检查你的提示是否可能不够具体。

## 以下是不应该做的：

**不要写整本小说**
冗长的提示包含无关信息可能会混淆模型或导致它专注于错误的事情。在你的描述中要简洁但完整。例如，在编码上下文中，你通常不需要以"你是一个世界级的程序员..."开头（一些人在一般ChatGPT使用中这样做，但对于编码任务，这通常是不必要的，可能会增加噪音）。

**不要假设AI会自己正确地填补细节**
如果某些事情很重要（如线程安全、特殊字符处理等），要提及。如果没有提及，假设AI可能不会处理它。

**当你需要确定性输出时避免开放式"创造性"提示**
例如，说"写一些代码来分析数据"可能导致AI猜测你想要什么分析。相反，要指定：

> 计算数字列表的平均值和标准差。

总之，*准确地说出你的意思*。AI越"知道"你真正想要什么，它就越能提供。如果你发现自己不得不多次纠正AI，问：我的初始提示是否可以更清晰？

# 迭代完善：与AI的反馈循环

即使有清晰的提示，你也不会总是在第一次尝试时得到完美的答案。将与AI的互动视为对话或迭代开发过程。这是我在第1章提到的反馈循环。

当AI给你代码时，要批判性地审查它，就像审查人类编写的代码一样。它是否满足要求？如果不，识别缺失或错误的地方。然后提供反馈或完善的提示。这可以在对话式AI中通过简单地继续对话来完成，或者在编辑器中通过为AI响应编写另一个注释来完成。

通过向AI提供反馈，你引导它更接近你想要的结果。在某种意义上，你正在为你的特定问题*训练*它。高级提示工程就像图2-2中的循环：提示→AI输出→审查→完善提示→AI输出→...直到满意。保持每次迭代的变更小是有用的；如果你过度修改提示，你可能会失去之前输出的一些好部分。

![图2-2. 高级提示工程循环。](images/chapter-02-figure-02-prompt-engineering-loop.png)

例如，你可能提示：

> 编写一个函数，接受整数列表并返回它们的和。

然后AI返回一个函数，但其代码假设非空列表，并且不能很好地处理空列表。然后你可以回复：

> 这看起来不错。但是，请修改它，如果列表为空则返回0。

然后AI会相应地更新函数。这样，你不必从头开始提示；你只是告诉AI进行调整。AI已经有之前给出的代码的上下文。

如果你使用内联助手，完善可能看起来像编辑代码，也许写一个注释如`# TODO: 处理空列表`，然后看AI是否建议修复。

另一个完善方法是在第一个输出不正确时用更多信息重新提示。假设你说"排序名字列表"，它给出了区分大小写的排序代码，但你想要不区分大小写。你可以重新表述：

> 不区分大小写地排序名字列表。

或者甚至：

> 之前的代码是区分大小写排序。修改它为不区分大小写。

在调试中，对于更复杂的逻辑错误（没有抛出明显的错误消息但输出错误），你可以提示AI逐步执行代码。例如：

> 逐行检查这个函数并跟踪每一步total的值。它没有正确累积——逻辑在哪里出错？

这是一个"橡皮鸭"调试提示的例子：你本质上是在要求AI模拟人类可能用print或调试器进行的调试过程。这样的提示经常揭示细微的问题，如变量没有重置或错误的条件逻辑，因为AI会拼出每一步的状态。如果你怀疑代码的某个部分，你可以放大：

> 解释这里的filter调用在做什么，以及它是否可能排除了比应该更多的项目。

让AI担任解释角色可以在解释过程中暴露错误。

解释后，直接询问你需要的内容通常是有效的：

> 什么可能导致这个问题，我如何修复它？

这邀请AI既诊断又提出解决方案。如果AI的第一个答案不清楚或部分有帮助，不要犹豫，问一个后续问题：

> 那个解释有道理。你能告诉我如何修复代码吗？请提供修正后的代码。

在聊天设置中，AI有对话历史，所以它可以直接输出修改后的代码。如果你在VSCode或Cursor中使用没有聊天的内联工具如Copilot，你可能而是在代码上方写一个注释：

```javascript
// BUG: 返回NaN，修复这个函数
```

不过，一般来说，交互式聊天产生更彻底的解释。

另一个后续模式：如果AI给出了修复但你不知道为什么，问：

> 你能解释为什么那个改变解决了问题吗？

这样，你为下次学习，并且你双重检查AI的推理是否合理。

LLM在示例和修正上蓬勃发展。如果你指出什么是错的或给出一个快速示例，AI可以纳入它：

> 如果输入是[]，它应该返回0，但现在它出错。

这个迭代过程是正常的。事实上，尝试将每个细节都塞进一个提示可能比几次来回交流效果差。利用这个优势。

在你的反馈中要有耐心和具体。不要说"不，那是错的"，而要说什么是错的或需要什么：

> 这个代码不能正确处理负数。它在总和中应该将它们视为0。

另外，如果AI偏离轨道，你可以引导它回来：有时重置或重新表述比试图挽救一个非常不正确的尝试更容易。使用你的判断。如果AI输出显示它完全误解了你，从头开始澄清你的提示。

当你完善时，你也会学习AI如何解释你的提示。这可以告知你如何编写未来的提示。你可能意识到，"哦，它把'登录系统'理解为整个UI。下次我会指定仅后端。"

把它想象成调试代码：如果AI输出是错的，"bug"可能在你的提示中，而不是在AI的处理中。就像当代码产生错误结果时你会检查和修复你的代码一样，当AI生成意外或不正确的输出时，你应该完善你的提示。你和AI之间的对话就像一个调试会话，每次交流都帮助你定位和修复问题。

# 比较两个提示

想象你有一个简单的Node.js函数，旨在将用户对象列表转换为按用户ID的查找映射。然而，它抛出一个错误。这是有错误的代码：

```javascript
// 有错误的函数：将用户数组转换为按ID的映射
function mapUsersById(users) {
  const userMap = {};
  for (let i = 0; i <= users.length; i++) {
    // 注意：使用<=会在最后一次迭代时越界
    const user = users[i];
    userMap[user.id] = user;
  }
  return userMap;
}

// 示例用法：
const result = mapUsersById([{ id: 1, name: "Alice" }]);
```

这个代码应该遍历用户数组并构建一个键是用户ID的对象。然而，有一个错误：循环使用`i <= users.length`而不是`i < users.length`。这将导致`users[i]`在最后一次迭代时为undefined，可能导致运行时错误（如尝试访问undefined的属性ID）。让我们看看AI助手可能如何响应关于这个问题的两个不同提示。

![图2-3. 提示质量影响比较：模糊提示产生通用或不正确的代码，而具体的、结构良好的提示生成准确的、上下文适当的解决方案。](images/chapter-02-figure-03-prompt-quality-comparison.png)

## 不好的提示

这是一个不好提示的例子：

> 为什么我的mapUsersById函数不工作？

AI对不好提示的响应（模拟）：

> 没有更多信息很难说。你确定函数是用适当的用户数组调用的吗？问题可能是users为空或不是数组，导致循环不能正确执行。也可能是在userMap上设置属性时有错误。确保每个用户都有id属性。如果users是undefined或null，那也会导致它不工作。

这个答案相当模糊。提示除了函数名外没有提供任何细节，所以AI在猜测通用原因（"可能数组为空或不是数组"，"确保每个用户有ID"）。它没有识别实际的错误。

这是当问题不够具体时得到的响应类型——模型只能重复常见的调试技巧。如果你是开发者，这不是很有帮助，因为它不解决真正的错误。提示中缺乏上下文和具体性产生了通用答案：本质上是AI版本的"你试过关掉再打开吗？"这与社区建议一致，即模糊提示导致无关或通用的答案。

## 改进的提示

让我们再试一次：

> 我有一个JavaScript函数`mapUsersById`，它应该将用户对象数组转换为按用户ID键控的`映射（对象）`。然而，当我运行它时它抛出错误。例如，当我传递`[ {id: 1, name: "Alice"} ]`时，我得到`TypeError: Cannot read property 'id' of undefined`。这是函数代码：
>
> ```javascript
> function mapUsersById(users) {
>   const userMap = {};
>   for (let i = 0; i <= users.length; i++) {
>     const user = users[i];
>     userMap[user.id] = user;
>   }
>   return userMap;
> }
> ```
>
> 它应该返回`{ "1": {id: 1, name: "Alice"} }`。错误是什么，我如何修复它？

AI对改进提示的响应（模拟）：

> 错误`Cannot read property 'id' of undefined`表明在某个点，user是undefined，然后你尝试访问user.id。查看你的循环，问题出在循环边界。你使用的是`i <= users.length`。如果`users.length`是`1`（数组中有一个用户），循环运行`i = 0`和`i = 1`。当`i`是`1`时，`users[1]`是undefined（因为数组中唯一的索引是0）。因此user在那次迭代中变为undefined，`user.id`抛出`TypeError`。修复是将循环条件改为`i < users.length`，这样`i`从`0`到`users.length-1`。例如：
>
> ```javascript
> for (let i = 0; i < users.length; i++) {
>   const user = users[i];
>   userMap[user.id] = user;
> }
> ```
>
> 这样，对于第一个（也是唯一的）元素，i将是0，然后循环停止。有了这个改变，你的函数将不再遇到undefined用户，应该按预期工作，为给定输入返回`{"1": {id: 1, name: "Alice"}}`。

这个响应直接指出错误（在循环中使用`<=`）并解释为什么它导致user为undefined。然后它提供正确的代码解决方案（`i < users.length`）并确认预期结果。改进的提示成功了，因为它遵循了我们的原则：它指定了语言（JavaScript），描述了函数的目的，包括了确切的错误消息和示例输入，甚至提供了有问题的代码片段。

注意AI如何使用错误消息作为线索来专注于循环边界——有针对性的提示使AI能够进行真正的问题解决，有效地模拟人类调试器如何思考："undefined可能从哪里来？可能来自循环索引。"这是详细提示好处的具体证明。
