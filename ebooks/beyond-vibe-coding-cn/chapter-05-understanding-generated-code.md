# 第5章 理解生成的代码：审查、完善、拥有

您已经学会了如何提示AI生成代码，此时您可能已经使用这些技术产生了一些代码。现在来了一个关键阶段：确保代码是正确的、安全的和可维护的。

作为开发人员，您不能简单地接受AI的输出并愉快地发布它。您需要审查它、测试它、可能改进它，并将其与您的代码库的其余部分集成。本章重点介绍如何理解AI给您的代码，迭代地编辑和调试它，并完全将代码作为您项目的一部分拥有。

本章涵盖：

- 根据您的原始意图解释AI的代码
- "多数解决方案"现象，或为什么AI生成的代码通常看起来像通用解决方案
- 审查代码以获得清晰度和潜在问题的技术
- 当AI编写的代码不如预期工作时进行调试
- 为风格或效率重构代码
- 编写测试以验证代码的行为

通过掌握这些技能，您将能够自信地将AI贡献集成到您的项目中。

# 从意图到实现：理解AI的解释

当您获得AI的代码时，您的第一步应该是将其与您的意图（您给出的提示）进行比较。代码是否满足您设定的要求？有时AI可能会稍微误解或只部分实现您的要求。

仔细阅读代码。在您的脑海中或在纸上逐步分析：

- 追踪它对典型输入的处理。
- 如果您的提示有多个部分（"做X和Y"），验证AI是否完成了所有部分。
- 确保AI没有添加您没有要求的功能——有时它会添加它"认为"有用的额外功能，比如添加日志或参数，这可能可以接受，也可能不可以。

就像您对同事的代码一样，如果不清楚某些内容，请记录下来。如果您寻找它存在的良好理由，您可能会找到一个。如果没有，查询它或考虑删除它。

例如，如果您要求一个素数检查器，AI代码还为每个数字打印类似"检查7…"的内容，这可能是您提示方式的产物或其训练数据中的模式（一些教程代码打印其进度）。如果您不想要那个，计划删除它或提示AI删除它。

还要确保边缘情况按您的预期处理。如果您打算让它处理空输入，它是否处理？如果输入可能是`None`或负数，AI是否考虑了这一点？

如果您的提示中有某些内容是模糊的，AI必须选择一种解释，请识别出这种情况发生的位置。也许您没有指定输出格式，它选择打印结果而不是返回结果。现在您必须决定是否接受或修改代码。

这个理解阶段至关重要；不要跳过它。即使您要测试代码，通过阅读理解它也是重要的，因为测试可能不会覆盖所有内容（对于一些明显的事情，阅读更快）。

最后，考虑AI的假设。AI经常倾向于"多数"或最常见的解释（这导致我们进入下一节）。

# "多数"问题：最常见不意味着最合适

在大量代码上训练的AI模型通常会产生在训练数据中最具代表性的解决方案（或最简单的合适解决方案）。我称之为*多数解决方案*效应。它在一般情况下是正确的，但对您的具体情况可能不是最好的。

例如，如果您在没有进一步上下文的情况下要求搜索算法，AI可能会输出基本的线性搜索，因为这是直接和常见的。也许您实际上需要二分搜索，但AI不知道效率是关键的，因为您没有这么说。线性搜索适用于许多中等情况，但如果性能是关键则不行。

类似地，AI可能使用全局变量，因为许多简单示例都这样做，但在您的项目中，这可能是不可接受的做法。

请注意AI的解决方案可能针对通用场景进行了优化。作为人类开发人员，您拥有AI缺乏的背景洞察力。

为了解决这个问题：

- 识别代码中的假设。如果它假设列表已排序或输入有效，那个假设可以接受吗？您指定了吗？如果没有，也许它应该包含一个检查。
- 考虑替代方案：如果您知道解决问题的多种方法（如不同的算法），AI是否选择了一种？是您想要的那个吗？如果不是，您可以提示替代方案或直接更改它。

如果AI代码在"通常"情况下工作，但对您重要的边缘条件不起作用，那是需要修复的事情。例如，也许它没有考虑某些数学中的整数溢出。在许多训练示例中，这可能没有被解决，但在您的上下文中，这可能是重要的。

理解AI倾向于通用解决方案将使您更好地审查其代码。这不是魔法或定制的；它是对解决方案的高度教育性猜测。定制是您的工作。

# 代码可读性和结构：模式和潜在问题

AI生成的代码通常有一些明显的模式。它可能：

- 包含比通常更多的注释或措辞奇怪的注释（因为它从教程代码中学习，这些代码往往有大量注释）
- 一致地使用某些变量名（如循环中的i、j、k）
- 以稍微冗长的风格布局代码（以覆盖一般情况）

检查这些并考虑它们是否符合您项目的风格。代码可能在功能上没问题，但需要可读性改进。在这个过程中，您可能想要：

- 重命名变量以更具描述性或与您的代码库一致。
- 删除或精炼注释。如果它在自解释的`if`语句上方添加了类似`# 检查数字是否为素数`的注释，您可以删除它。但如果它有解释复杂逻辑的注释，那很好——保留或改进它。
- 通过将代码通过linter或formatter（如Python的Black或Go的`gofmt`）运行，确保一致的格式化，以匹配您想要的间距和括号样式。

还要注意任何不寻常的结构。AI是否定义了多个类或函数，而您期望一个？有时它可能会将问题分解为多个函数，因为训练示例就是这样做的。如果那是过度工程化，您可以内联它们（反之亦然）。代码是否太聪明或太天真？AI有时会产生非常直接的解决方案，或者偶尔产生花哨的单行代码。这是否符合您团队的偏好？如果不相应调整。

其他需要注意的潜在问题包括：

**差一错误**
是的，AI也会犯这些错误。例如，循环边界可能很棘手。如果您有时间，在心理上通过循环测试一个简单情况。

**未处理的异常**
代码是否假设文件成功打开或所有输入格式正确？如果需要，添加错误处理。

**性能陷阱**
也许AI正在对大型数据集使用内部循环进行成员检查，即使存在更好的方法，比如使用集合。AI解决方案可能正确但不是最优的。

**库使用**
如果代码使用库，确保它是您想要使用的库（并且可用）。有时它可能会使用，比如`numpy`进行简单求和（因为它在其训练数据的示例中看到过）。如果引入该依赖不值得，您可以切换到纯Python或您想要的库。

**不一致**
偶尔，AI代码可能有轻微的不一致，比如函数文档字符串说一件事但代码做另一件事（例如，如果它修改了逻辑但没有修改注释）。修复那些。

**轻微语法问题**
这在经过良好测试的模型中很少见，但在可能混淆某些东西的语言中并非不可能。

**使用过时的API**
AI可能使用库函数的旧版本，该版本已经改变。例如，如果您看到不认识的函数调用，快速检查库文档以确保它对您使用的版本是正确的。

**占位符**
如果AI输出使用类似"Your code here"的占位符（很少见，但在通用模板中可能发生），请填写这些。

简而言之，将AI代码视为实习生写的代码然后离开了。您需要审查其质量并正确集成它。

# 调试策略：发现和修复错误

假设您运行代码（或为其编写测试，我们很快会涵盖）并且某些内容不起作用。调试AI生成的代码与调试您自己或他人的代码没有区别——除了您没有编写它，所以您可能不太熟悉。但因为您已经仔细阅读了它，您的情况很好。

这是一个六步调试方法：

1. **重现问题。**
   使用失败的输入运行函数或代码。观察输出或错误。

2. **定位问题来源。**
   使用典型的调试技术，如打印语句，或使用调试器逐步执行。如果是逻辑错误（错误输出），手动或通过打印追踪逻辑，看看它在何处偏离您的期望。

3. **检查提示与代码。**
   有时错误只是代码没有完全实现要求，比如您要求某些东西排序但它没有正确排序。这可能意味着AI的逻辑有缺陷，或者边缘情况（如空列表）没有处理。

4. **利用AI调试！**
   您实际上可以将有问题的代码重新输入AI并说，"这段代码在输入X上给出错误结果。您能帮助发现错误吗？"通常，它会分析它（像代码审查）并指出问题。例如，也许它看到循环应该到`len(arr)`但到`len(arr)-1`。它可能更快地捕捉到这一点。（注意不要完全信任它——但这就像请同事帮助调试。）

5. **修复代码。**
   现在您有选择：手动修复或提示AI进行更正版本。如果修复很明显，就做吧。如果不是，您可以尝试类似"上述函数在输入X上失败（期望Y，得到Z）。请更正它。"然后AI可能会相应地调整代码。

6. **再次测试。**
   确保错误已解决，并且没有引入新问题。

我建议使用测试驱动的调试。如果可能，为关键函数编写一些测试（本章后面的测试部分会详细介绍）。任何失败的测试将直接显示错误。对于除了最简单函数以外的任何内容，这可能比手动检查更快。

最后，调试时，确保您问*为什么*，而不仅仅是*什么*。尝试理解为什么AI犯了错误。提示在那个点上不清楚吗？这可以告知您下次如何提示或是否需要始终在AI输出中双重检查那个方面。例如，如果您注意到AI经常不处理空输入，除非被告知，您将开始在提示中始终指定这一点并审查它。

# 为可维护性重构：使AI代码成为您的代码

一旦代码在功能上正确，考虑重构它以符合您项目的标准，并使其在未来更容易使用。AI的工作是快速给您代码；您的工作是完善它。

这是另一个六步过程，这次用于重构：

1. **与风格指南对齐。**
   将代码通过您的formatter或linter运行。修复任何警告，如"变量名应为小写"或"行太长"。这立即使代码看起来像代码库的其余部分。许多AI工具在风格方面做得不错，但可能需要轻微调整。

2. **改进命名和结构。**
   如果AI在类中命名了函数`_helper1`和`_helper2`，而您更喜欢有意义的名称，请重命名它们。如果它创建了一堆只使用一次的小函数，也许内联它们，除非它们增加了清晰度。

3. **删除任何不必要的部分。**
   例如，也许AI在输出中包含了您没有要求的主块或测试代码。如果您不需要那个，删除它。相反，也许它将所有内容写在一个函数中，但您想将其拆分为更小的部分以获得清晰度；如果是这样，现在进行拆分。

4. **添加文档。**
   如果此代码旨在成为库或其他人将使用的模块，在适当的地方添加文档字符串或注释。AI可能已经注释了一些内容，但确保它符合您的标准。例如，也许您的项目需要记录参数和返回值的特定文档字符串格式。

5. **如果需要，进行优化。**
   现在代码工作了，它足够高效吗？如果此代码可能在紧密循环中调用或在大型数据上调用，检查其复杂性。AI可能没有使用最优方法（再次，"多数解决方案"可能是简单循环，而不是更优化的方法）。如果性能是问题，重构为更好的算法。您可以再次涉及AI：

   > 通过使用集合而不是列表进行查找来优化此代码以运行更快。

   但作为开发人员，您通常知道想要什么模式，所以您可能只是实施该更改。

6. **如果需要，简化。**
   有时AI代码可能过于冗长。例如，它可能在应该使用单个条件返回的地方使用if-else与返回。虽然显式代码不一定坏，但您可能希望将其简化为更少的行以提高可读性而不失去清晰度。

重构的目标是，如果另一个开发人员后来拉起这段代码，不应该明显看出"AI写了这个"。它应该看起来就像好代码。这通常意味着给它那些使代码干净的小小人性化修饰。

当您重构时，您需要验证您没有破坏任何东西。所以让我们转向测试。

# 测试的重要性：单元、集成和端到端

测试总是重要的，但对AI生成的代码特别重要，原因有两个。首先，因为您不是从头编写的，您希望确保它在所有情况下都能工作。其次，如果您稍后提示AI进行更改或集成更多AI代码，测试帮助您确保任何新更改不会破坏现有功能。让我们快速看看不同类型的测试：

**单元测试**
为从AI获得的每个函数或模块编写测试，特别是覆盖边缘情况。对于我们的素数示例，您可能用素数、非素数、1（边缘情况）、0或负数（可能定义预期行为）、大素数等进行测试。如果代码通过了所有这些测试，它可能是正确的。

您甚至可以要求AI生成这些测试：

> 为上述函数编写PyTest单元测试，覆盖边缘情况。

它通常做得不错。仍然审查它们以确保它们有效并覆盖您认为必要的内容。

**集成测试**
如果AI代码与代码库的其他部分交互，比如使用数据库的函数，编写在上下文中调用它的测试。它是否实际将应该的内容存储到数据库？如果它产生被另一个函数消费的输出，在测试中将它们链接起来。

**端到端测试**
如果此代码是更大工作流程的一部分，从头到尾运行场景。例如，如果AI代码是web路由的一部分，在测试环境中对该路由进行测试请求，看看格式、错误处理和其他所有内容是否都保持不变。

您需要进行的测试级别取决于代码的关键性和复杂性。但即使是快速的手动测试运行或脚本中的简单断言语句也比没有验证好。记住，测试不仅仅是发现错误；它还锁定行为。如果您稍后更改某些内容（或AI这样做），测试帮助您确保代码功能不会回归。

测试也是断言所有权的好方法。一旦您测试并修复了任何问题，您可以对代码有信心。此时，可以说代码是"您的"，就像代码库中的任何其他代码一样。您理解它，信任它，并有测试来保护它。

## 关于AI和测试的说明

一些AI编码工具开始集成测试建议。例如，CodeWhisperer有时会在一段代码后建议断言。使用这些建议作为起点，但不要假设它们100%全面。想出创意性的边缘情况——这是人类直觉仍然非常有价值的地方。

# 总结和下一步

我们已经经历了生成、理解、调试和重构代码。这个循环可能在短时间内发生（对于小函数，几分钟内）或需要更长时间（对于复杂模块，几小时或几天，有间歇性AI协助）。

重要的是承认*您，开发人员，对最终代码负责*。AI是加速创作的工具，但如果某些内容失败，它不会承担责任。还有许可或版权风险：一些AI提供商说，超过一定长度的输出可能在统计上可能包含复制的材料。这很少见，提供商大大缓解了这个问题，但就像您扫描Stack Overflow答案寻找任何明显许可的文本或归属一样，进行快速检查——特别是如果输出很大或太干净。例如，如果您提示"实现快速排序"，AI给您20行干净的代码，那可能没问题，是常识。但如果您要求一些晦涩的内容并获得大量代码，尝试在线搜索其中的唯一字符串，看看是否是从某处逐字复制的。这个问题最近变得更加明显，有记录的案例显示AI系统复制期刊文章和其他受版权来源的文本。作为负责任的代码所有权的一部分，开发人员应该验证任何看起来超出通用模式或似乎异常特定于特定来源的AI生成内容的来源。

最后，将代码集成到您的项目中：将其添加到您的版本控制系统，也许在提交消息中提到AI帮助。没有要求这样做，但一些团队喜欢跟踪它。

随着时间的推移，随着需求变化，您可能会修改此AI生成的代码。像对待任何其他代码一样对待它：不要想，"哦，那是AI的代码；我会要求AI更改它。"您可以，如果您想，但您也可以自由地手动修改它。做最有效和可维护的事情。

通过仔细的审查和测试，AI生成的代码成为您项目中的更多代码。在那时，无论是AI写了第10行还是您写的都无关紧要——重要的是它满足项目的需求和标准。

通过遵循这些实践，您利用AI编码的速度同时确保质量。您避免了盲目信任AI输出的陷阱，而是将其集成到专业的开发工作流程中。
