# 第6章 AI驱动的原型：工具和技术

本章探讨了AI驱动的氛围编码如何加速软件开发的原型阶段。原型设计的全部目的是快速将想法转化为工作模型。有了AI助手，开发人员可以在几小时内完成通常需要几天的工作，快速迭代概念。我将讨论使用AI从概念到原型的技术，比较流行的AI原型工具（包括Vercel v0和截图到代码工具），并研究如何在AI指导下迭代完善原型。我还解决了将粗略的AI生成的原型转换为生产质量代码的关键步骤。在本章中，我还将查看AI驱动原型导致成功结果的案例研究，并展示这种方法的潜力和陷阱。

# 使用AI助手进行快速原型设计

原型设计大大受益于AI生成代码的速度。原型设计的目标不是精美的、生产就绪的代码，而是您可以评估和改进的概念证明。AI编码助手通过从最小输入快速生成功能代码在这里大放异彩。例如，开发人员不是手动编码UI模型，而是可以用自然语言描述所需界面，让AI生成HTML/CSS或React组件。这允许非常快速的创意到实现周期。

一位用户体验工程师描述了使用生成AI工具如何完全改变了他的工作流程："我能够使用生成AI生成功能原型的速度令人惊叹。我在几小时内构建了原型，[这]通常需要几天。"AI自动处理常规的样板代码和重复模式，让他专注于更高级的设计决策。在原型设计中，这意味着您可以在相同的时间内尝试更多想法。另一个好处是AI几乎可以立即填充原型的繁琐部分（基本UI组件、表单处理、示例数据），让人类开发人员专注于被测试的核心概念或独特功能。

然而，重要的是记住原型的代码通常是可抛弃的代码。AI可能快速生成工作解决方案，但该代码可能没有为可维护性或规模而构建。这在原型中是可以接受的，因为速度和实验比优雅更重要——只要您计划为生产重构或重写关键部分（我将在本章后面讨论这个过程）。正如第3章和第4章所探讨的，AI可以自信地处理原型大约70%的编码工作，而您指导整体架构并纠正任何关键缺陷。

有意编程，第1章介绍的概念，在这里是必不可少的。您提供高级指令（您希望原型做什么或看起来如何的意图），让AI填充实现。为了说明，让我们使用AI助手完成一个非常简单的原型示例。假设您有一个新服务注册页面的想法。您想快速原型化前端，所以您从以下开发人员提示开始：

> 创建一个简单的HTML页面用于注册表单，包含姓名、电子邮件和密码字段，以及提交按钮。使用基本样式使其看起来体面。

像Cline这样的现代代码助手可能会提供完整的HTML/CSS示例：

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>Sign Up</title>
  <style>
    body { font-family: Arial, sans-serif; max-width: 400px; margin: 2em auto; }
    form { display: flex; flex-direction: column; }
    label { margin-top: 1em; }
    input, button { padding: 0.5em; font-size: 1em; }
    button { margin-top: 1em; background: #4CAF50; color: #fff; border: none; }
  </style>
</head>
<body>
  <h2>Sign Up</h2>
  <form>
    <label>Name: <input type="text" name="name" required></label>
    <label>Email: <input type="email" name="email" required></label>
    <label>Password: <input type="password" name="password" required></label>
    <button type="submit">Create Account</button>
  </form>
</body>
</html>
```

这个AI生成的代码是注册表单调用的一个不错的起点。它包括基本布局和一些用于样式的内联CSS。作为开发人员，您不必手动编写任何HTML/CSS——您只需描述结果，AI就提供实现。这展示了有意编程：您专注于UI应该做什么，AI弄清楚如何做。

当然，这是一个微不足道的例子。在实践中，原型可能更复杂，涉及多个文件、框架或数据。但原则是相同的：您使用AI快速创建初稿。然后您可以在浏览器中打开这个原型，看看它的外观，并进一步完善。这引出了下一个主题：使用超越纯语言提示的专业AI原型工具。

# AI原型工具

虽然像Gemini、ChatGPT和Claude这样的通用助手可以从提示生成原型代码，但专业AI辅助原型工具的格局继续快速发展。在我撰写本文时，可用工具在AI生成原型中保真度和控制之间基本权衡的不同方法。

生态系统已经成熟，以满足不同的原型需求。一些工具擅长将视觉设计转换为代码，允许设计人员上传截图或草图，并在几秒内接收工作HTML、CSS或React组件。这种"截图到代码"能力大大加速了设计到代码的过程，当您有手绘草图或需要快速实施的Figma设计时特别有价值。像Vercel v0这样的工具体现了这种方法，提供对原始设计的高保真度，同时牺牲一些对代码结构的控制。

其他平台专注于通过对话界面生成完整应用程序。这些允许用户用自然语言描述功能并接收完整的堆栈实现。例如，想要快速应用程序而不编码的设计人员可能会使用像Lovable或Bolt.new这样的工具，它们提供可以搭建整个应用程序的高级提示界面。一些设计人员报告在几小时内构建功能原型，传统上需要几天，AI自动处理繁琐的组件和模式。

第三类将AI直接集成到开发环境中，在原型设计阶段作为智能结对程序员。这些AI增强的IDE如Cursor、Windsurf和Cline允许开发人员在代码生成过程中保持更多控制，同时仍然受益于AI加速。区别通常在于工作流程哲学：一些优先考虑自动化应用更改以进行快速实验，而其他需要明确接受修改以进行更仔细的迭代。

所有这些方法的共同点是它们能够压缩从概念到工作原型的旅程。然而，它们在两个关键维度上显著不同，这些维度塑造了它们对不同用户和用例的实用性。

**保真度**指的是生成的输出与您的输入或意图的匹配程度。截图到代码工具通常提供对视觉设计的高保真度，但可能产生与您架构偏好不符的代码。对话工具可能更宽松地解释您的要求，生成功能但通用的实现，需要完善。

**控制**涵盖了您指导和修改生成过程的能力。一些工具作为产生完整输出的黑箱操作，而其他允许通过持续对话或直接代码编辑进行迭代完善。当您需要特定的架构模式、性能优化或与现有代码库集成时，这个维度变得至关重要。

理解这些权衡有助于为您的原型需求选择正确的工具。验证新交互模式的设计师可能优先考虑保真度和速度，接受对实现细节的较少控制。探索技术可行性的开发人员可能重视控制和透明度，即使这意味着在生成过程中更多的手动干预。

这些工具都不能在没有人工监督的情况下产生生产质量的代码。它们通常提供我所谓的"80%原型"——足以测试概念并向利益相关者演示，但需要额外的工作才能生产部署。剩下的20%通常涉及安全强化、性能优化、错误处理和架构完善。

即使在快速原型设计期间，快速代码审查仍然必不可少。虽然您可能不会在原型中完善每个细节，但扫描明显问题如暴露的API密钥或不安全的数据处理可以防止问题传播到后来的开发阶段。大多数现代工具提供对其生成代码的透明度，允许您检查和理解正在创建的内容。

随着AI原型格局继续发展，特定工具无疑会改变，但这些保真度与控制的基本考虑，以及对人工监督的需要，将保持不变。关键是理解您的原型目标并选择符合您特定需求的方法，无论是快速视觉实施、功能演示还是技术探索。

# 从概念到原型：迭代完善

AI驱动原型的优势之一是迭代循环：您可以生成初始版本，然后通过与AI交互来完善它。您不必手动编辑代码，只需告诉AI您想要什么改变（见图6-1）。虽然我提倡比纯粹的"即兴"氛围编码更负责任的方法，但快速反馈循环绝对是原型中应该接受的东西。

![图6-1. 迭代原型完善过程：初始提示生成基线原型，开发人员反馈驱动连续改进，通过AI协作创建日益完善的解决方案。](images/chapter-06-figure-01-iterative-prototype-refinement-process.png)

大多数AI原型工具都保留您请求的历史或上下文，这非常有用。这意味着AI记住您应用程序的目的和先前的指令，因此您不必每次都重新解释所有内容。这种上下文持久性是氛围编码环境的标志：与AI的对话成为开发日志。

典型的迭代完善过程如下：

## 第1步：初始生成

您提供提示或输入来创建原型：

> 生成一个基本的费用跟踪应用程序，带有添加费用的表单和列出费用的表格。

## 第2步：审查和运行代码

您获得生成的代码并运行它。也许它工作，但您注意到一些可以改进的地方。例如，UI功能正常但平淡，或者表格不排序费用。

## 第3步：完善您的提示

您回到AI并提供额外指令。例如：

> 使费用列表可以按金额或日期排序。

AI可能会修改代码以包含排序逻辑或使用可排序表格的库：

> 添加一些颜色样式，也许使用现代CSS框架。

AI可以集成CSS库（如Tailwind或Bootstrap）或只是添加自定义样式使其看起来更好：

> 验证表单，这样您不能添加没有名称和金额的费用。

AI可能会添加简单的前端验证。

这些提示中的每一个都修改原型。因为AI理解上下文（像Cursor和持续聊天工具这样的工具将保持代码状态），它通常可以在正确的位置应用更改——例如，在表单中插入验证代码或重写表格渲染以包含可排序列。

## 第4步：重复

每次完善后，您检查结果。如果AI引入了新问题或没有完全按照您的意图执行，您通过提示澄清或修复它：

- 排序是反的——请默认按升序排序。
- 新的颜色方案很好，但使标题深蓝色而不是黑色。

每个迭代循环都相当快——通常只需要几秒钟的处理——这意味着您可以在一小时内进行十几次迭代。与手动编码和检查所有这些更改相比，AI方法可以显著更快。对于广泛更改，如重新样式化或添加功能，尤其如此。

重要的是，与AI迭代需要清晰的沟通。这是您的提示工程技能发挥作用的地方。您对想要的变化越明确和清晰，AI就越可能正确地执行。例如，说"让它看起来更好"是模糊的。更具体的提示给助手具体的方向：

> 应用带有蓝色标题的浅色主题并增加字体大小以提高可读性。

如果您使用像Vercel v0或Lovable这样的工具，您甚至可以使用其界面指向UI的一部分并说"将其更改为X"。

让我们看一个开发人员和AI编码助手之间的简短对话迭代过程：

> 生成一个待办事项列表的React组件。它应该显示任务列表和添加新任务的文本输入。

AI生成一个带有任务状态的React组件，并渲染列表、输入框和添加任务的按钮：

> 很好。现在使每个任务可编辑并添加复选框标记完成。

AI更新代码，为每个任务渲染复选框和编辑功能，并相应地修改状态。开发人员运行应用程序，发现编辑工作，但没有删除按钮：

> 为每个任务添加删除按钮。

AI在代码中添加删除功能：

> 布局有点拥挤。添加一些基本的CSS来间隔列表项并使字体稍大。

AI输出CSS样式，可能是组件中的`<style>`或单独的CSS片段：

> 看起来更好！

这种来回可能继续，直到原型符合愿景。最终，开发人员获得一个待办事项列表应用程序的工作原型，具有创建、编辑、完成和删除功能——所有这些都通过自然语言请求和快速AI代码输出构建。

在整个过程中，记住开发人员仍然是发生什么的导演。AI可能提出实现功能的方法，但您决定这是否符合您的需求。有时AI的实现是正确的，但不是您期望的（也许它使用与您心中不同的UI方法）。您可以接受它（如果它不损害原型目标）或指示AI更改为您喜欢的方法。

# 将原型发展为生产

原型旨在成为概念证明和学习什么有效的工具。一旦它达到了那个目的——比如您已经用用户验证了设计或证明某个功能是可行的——下一步通常是将其转变为生产应用程序。这个过渡是一个关键的转折点。AI仍然可以提供帮助，但人类开发人员必须打磨原型的粗糙边缘。本节讨论从原型到生产代码时的一些关键考虑因素。

首先，仔细审查架构和代码结构。原型可能在内部很混乱。也许您所有的代码最终都在一个文件中，或者您为了速度绕过了某些最佳实践。现在是引入适当结构的时候。例如，如果原型是单页脚本，您可能将其分离为多个模块；对于Web UI，您可能引入适当的组件结构；对于后端，您可能设置正式的模型-视图-控制器（MVC）架构模式。

虽然AI编写了大部分原型，但作为开发人员，您最了解架构目标。您甚至可能启动一个新项目并使用原型作为参考或脚手架，也许重用一些原型代码，但通常将其视为可抛弃代码。其他人可能逐步重构原型代码库，AI建议重构或生成测试以确保在清理过程中不会破坏任何内容。

接下来，添加错误处理和边缘情况。原型代码通常专注于晴天场景，但如果API调用失败怎么办？如果输入为空怎么办？系统地遍历每个功能并考虑潜在的失败模式。

AI可以帮助您集思广益边缘情况，给出这样的提示：

> 这个功能的潜在错误情况是什么以及如何处理它们？

助手可能会列出一些场景（网络错误、坏输入、并发问题），您可以为其实现处理（或要求AI帮助实现）。确保代码的稳健性是使其生产就绪的一部分。

您的原型代码可能没有优化，所以检查任何效率低下或可能构成安全问题的部分。例如，也许原型中的AI使用了一个在小型测试数据集上工作的朴素算法，但对于真实数据会很慢。识别这些点并优化它们。（我将在第8章介绍常见的AI生成代码缺陷。）

一个策略是在原型上运行性能测试或使用分析器查看瓶颈，然后要求AI帮助优化该功能。绝对还要审查安全功能如身份验证和数据处理——AI原型使用没有正确参数化的SQL查询（冒SQL注入攻击风险）或包含敏感信息并不罕见。这些问题必须修复。2021年的一项研究发现，大约40%的AI生成的代码有潜在漏洞。所以生产化的一部分是保持警惕。手动运行静态分析和/或安全测试，或提示AI"扫描此代码的安全问题"。

原型通常缺乏文档，您需要在正式化代码时添加这些：每个模块的清晰的人工审查解释将帮助未来的团队成员，以及您，当您几个月后重新访问代码时。一旦您清理了代码，您可能会提示AI工具基于描述系统如何工作的代码生成Markdown API文档或README。第1章讨论了AI如何产生代码解释；这是利用这一点的好时机。

正如您在第5章中学到的，彻底测试您的原型至关重要。您可能为核心逻辑编写单元测试，为主要流程编写集成测试等。您可以通过要求AI生成测试用例来加速这个过程：

> 为待办事项列表组件编写Jest测试，涵盖添加、编辑、完成、删除任务。

然后运行并调整它生成的测试。拥有良好的测试套件使您在重构原型代码时有信心。

有时您可能决定完全替换代码的某些部分——例如，如果原型使用了一些快速而肮脏的库或不适合长期使用的hack。AI也可以加速这一点。假设您的原型代码使用本地数组存储数据，但现在您需要适当的数据库集成。您的提示可能类似于：

> 集成SQLite数据库用于存储任务，而不是内存数组。

AI可以为这种集成提供起点，然后您应该完善它。

在进行这些更改时，明智的做法是将您的心态从"快速原型模式"切换到更规范的工程方法。AI现在是您改进代码质量的助手——它不再只是吐出快速功能。动态有点不同：现在稳定性和质量是您的首要任务，您可能更批判性地评估每个AI建议。正如我在第4章后面提到的，高级开发人员可以从AI获得巨大好处，因为他们知道接受什么和修复什么。在这个阶段，您将大量运用那种高级思维：您对最终系统有愿景，因此您让AI负责特定的改进或实施。

为了具体讨论这个，让我们考虑一个简短的例子。想象一个独立开发人员Jane，她想构建一个小型Web应用程序，将CSV文件中的数据转换为图表。她使用AI助手在仅仅一个周末内完成快速原型：一个带有API的基本Node.js脚本，加上一个简单的前端来上传CSV并使用JavaScript图表库渲染图表。

她向一些潜在用户演示这个原型并获得积极反馈，因此Jane决定将其转变为真实产品（Web服务）。以下是她导航过渡的方式：

## 加固后端

原型的Node.js API没有身份验证（任何人都可以上传数据）。对于生产，她需要用户账户和身份验证。她使用AI集成身份验证系统（可能基于JWT）。AI提供脚手架，但她仔细审查它以确保密码正确哈希和令牌安全。她还向上传端点添加输入验证（AI没有做那个），使用AI建议的代码和她自己的调整的组合。

## 重构前端

初始前端是单个HTML文件，脚本标签指向CDN的依赖项。Jane决定重构为结构化的React应用程序以获得可维护性。她首先要求AI通过使用构建系统和npm而不是脚本标签来重构她的项目使其更生产就绪。然后她要求AI帮助将它们集成为React组件。例如，它将原型中的图表渲染代码转换为`<Chart>`组件。Jane使用AI加速编写这些组件，但她确保状态管理和组件层次结构遵循最佳实践（原型没有深入考虑的事情）。

## 测试和检查性能

Jane为核心功能（CSV解析、数据转换）编写单元测试。当她不确定边缘情况时，她查询AI：

> 我应该为CSV解析测试哪些边缘情况？

它建议场景如空字段和不规则列，她将这些纳入测试中。她还注意到原型将整个CSV文件加载到内存中；对于大文件，这可能导致崩溃。她修改代码以流处理，并使用AI双重检查她的流逻辑。现在应用程序可以更可靠地处理更大的文件。

## 完善UI

原型UI是功利性的。对于她的产品，Jane在用户体验上花费更多时间。她要求AI推荐响应式布局并可能集成CSS框架。AI添加Bootstrap，然后她使用它来改善外观（表单、按钮、布局）。她之后手动微调一些CSS。这个完善阶段较少关于繁重编码，更多关于设计选择，但AI仍然通过提供标准UI模式的快速代码（如导航栏和加载旋转器）来帮助。

经过这些努力，曾经粗糙的原型是一个更清洁、更安全和可扩展的应用程序，准备为真实用户使用。Jane部署它，感到自信，因为她添加了测试并审查了AI生成的代码。从原型到生产的过程可能需要她几周时间，而从头编写整个产品需要更长时间。AI加速了初始原型并继续协助过渡，但Jane的人工监督和重构在达到生产质量方面是不可或缺的。

# 解决AI原型中的挑战

虽然AI驱动的原型设计很强大，但并非没有挑战。作为开发人员，您应该意识到这些并知道如何减轻它们。两个特别感兴趣的领域是范围蔓延和集成。

因为使用AI添加功能如此容易，您可能被诱惑继续添加"一个更多东西"到原型，这种现象称为范围蔓延。这可能导致一个不断增长的原型试图成为最终产品。记住原型的目的：专注于您想要回答的关键问题或要演示的核心体验。如果您发现自己实施登录系统、支付处理等，问这真的是否在原型阶段需要。最好存根这些（AI可以生成假的登录流程，不是真实的，只是为了模拟）。保持原型专注将节省您的时间，并使其更容易在以后抛弃或重做。

## 保持专注

写下您原型的目标（"演示用户可以上传CSV并获得图表以测试可行性"），并以此作为北极星。使用AI快速达到该目标，抵制为原型镀金的诱惑。

其次，存在与真实系统集成的问题。原型通常使用模拟数据或简化的子系统。如果您的AI原型使用虚拟数据或本地文件，在生产中将其与真实数据库或服务集成可能是非平凡的。在原型设计时要意识到采取了一些捷径。例如，也许原型电子邮件实际上没有发送，只是记录到控制台。在生产中，您需要真实的电子邮件服务。AI可以稍后帮助集成这些，但最好保持跟踪：在原型设计时维护一个"如果我们继续前进要解决的事情"列表。这样您不会忘记哪些部分是临时的。如果在团队中工作，明确沟通这些。例如，您可能在代码中留下注释：`// TODO：在这里集成真实的电子邮件服务`。许多AI工具在生成简化解决方案时实际上自己包含这样的TODO注释，这很有帮助。

通过预见这些挑战，您可以有效地使用AI原型而不会陷入其陷阱。当经过深思熟虑使用时，结果是一个在创纪录时间内开发的稳健原型，准备要么转变为最终产品，要么在提取其提供的教训后被搁置。

# 总结和下一步

在本章中，您看到了AI辅助的氛围编码如何加速原型过程。通过让AI处理代码生成的繁重工作，开发人员可以以前所未有的速度从概念转向工作模型。我涵盖了用于UI生成的Vercel v0、用于全栈原型的Lovable以及AI增强的IDE如Cursor和Windsurf等工具——每种都启用了快速原型的不同方面。我还强调了AI原型的迭代性质：在快速循环中生成、测试和完善，自然语言提示指导变化。

虽然AI驱动的原型可以在几小时内产生功能演示，但我们也讨论了向生产的关键过渡。信息很明确：原型不是最终产品。它是初稿。人类开发人员必须重构和加固代码，AI继续协助那个旅程（建议改进、生成测试等）。使用这些技术的个人和团队的案例研究突显了真实的生产力收益——几天而不是几周构建的原型，实现更快的用户反馈和业务决策。

到现在，您应该欣赏氛围编码如何使原型设计感觉更像与助手头脑风暴而不是费力制作样板代码。这是一种根本不同的氛围：更具对话性，更高级，快得多。然而，您也看到了即使在快速原型中保持代码质量意识的重要性——当将其发展为超越原型阶段时绝对如此。

在第7章中，我将重点从快速原型转向使用AI协助进行全面Web应用程序开发。虽然原型探索可能性，但全规模开发需要系统性的架构、实施和部署方法。
