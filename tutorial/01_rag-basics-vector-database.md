# RAG 基础 & 向量数据库

这篇教程面向刚接触 RAG 和向量数据库的读者。读完后，你应该能回答三个问题：

1. RAG 为什么需要向量数据库？
2. 向量数据库和传统数据库到底有什么不同？
3. 如何用熟悉的“数据库 + 应用程序”思路理解“向量数据库 + LLM”？

本文结合本项目 `rag-demo` 的 5 个 demo，介绍 RAG 中数据摄取、切分、索引、检索、后处理和回答生成的基础概念。

## 1. 先理解 RAG 要解决的问题

大语言模型本身很强，但它有几个明显限制：

- 它不天然知道你公司内部文档、项目规范、工单、会议纪要里的内容。
- 它的训练知识有时间边界，不能自动知道刚更新的业务资料。
- 它生成答案时可能“看起来合理但事实不对”。

RAG 是 Retrieval-Augmented Generation 的缩写，通常译为“检索增强生成”。它的核心思想很简单：

```text
先从你的知识库里找相关资料，再把资料交给 LLM 生成答案。
```

所以 RAG 不是让模型“背下所有私有知识”，而是在回答问题前，先做一次检索，把相关上下文塞进 prompt。

一个典型 RAG 流程如下：

```text
原始文档
  -> 加载和清洗
  -> 切分成 chunk
  -> 用 embedding 模型转成向量
  -> 存入向量数据库
  -> 用户提问时也转成向量
  -> 在向量数据库中找相似 chunk
  -> 把检索结果交给 LLM
  -> LLM 生成最终回答
```

在很多 RAG 框架中，这条链路也会被抽象为：

```text
Document -> Node/Chunk -> Index -> Retriever -> Response Synthesizer
```

本项目用更工程化的方式把它落到 CLI demo 中：

```text
Markdown files -> Loader -> Splitter -> Embedding -> Vector Store -> Retriever -> LLM QA
```

## 2. 什么是 embedding 和向量

向量可以先理解成“一串数字”。

例如一句话：

```text
“如何申请年假？”
```

embedding 模型会把它转换成类似这样的数字数组：

```text
[0.12, -0.08, 0.34, ...]
```

真实向量通常有几百到几千个维度。每个数字单独看没有业务含义，但整组数字表达了文本的语义位置。

如果两段文本意思接近，它们的向量距离通常也更近：

| 文本 | 语义关系 |
| --- | --- |
| 如何申请年假？ | 和“休假流程”接近 |
| 年假审批需要谁同意？ | 和“休假流程”接近 |
| Kubernetes 如何配置 Ingress？ | 和“休假流程”较远 |

这就是向量检索的基础：不只按关键词匹配，而是按语义相似度查找。

## 3. 向量数据库是什么

向量数据库是专门用来存储、索引和检索向量的数据库。它通常会同时保存两类数据：

| 数据 | 例子 | 用途 |
| --- | --- | --- |
| 向量 | `[0.12, -0.08, 0.34, ...]` | 做相似度检索 |
| 元数据 | 文件名、标题、部门、时间、权限标签 | 做过滤、展示和溯源 |

向量数据库常见能力包括：

- 高维向量存储。
- Top-K 相似度检索。
- metadata filter，例如只检索某个部门、某个时间范围、某种文档类型。
- 近似最近邻索引，例如 HNSW、IVF 等，用来提升大规模检索性能。
- 持久化和服务化 API，方便应用程序调用。

本项目中涉及的向量数据库或向量存储包括：

| 组件 | 在本项目中的位置 | 适合理解的点 |
| --- | --- | --- |
| Chroma | `rag-demo t1` | 本地 RAG Hello World |
| Milvus | `rag-demo t2` | 百万向量压测、索引参数 |
| Qdrant | `rag-demo t3` | payload filter、HTTP API |
| Weaviate | `rag-demo t3` | schema-first、GraphQL 查询 |

## 4. 向量数据库和传统数据库的区别

传统数据库最擅长回答“精确条件查询”：

```sql
SELECT * FROM documents
WHERE department = 'HR'
  AND title LIKE '%年假%';
```

向量数据库最擅长回答“语义相似查询”：

```text
给我找出和“入职后怎么休假”意思最接近的 5 段内容。
```

两者不是谁替代谁，而是解决的问题不同。

| 对比项 | 传统数据库 | 向量数据库 |
| --- | --- | --- |
| 主要数据 | 行、列、JSON、关系 | 向量、文本片段、元数据 |
| 查询方式 | 精确匹配、范围查询、JOIN、聚合 | 相似度搜索、Top-K、混合过滤 |
| 典型问题 | 某个订单金额是多少？ | 哪些文档和这个问题语义最相关？ |
| 索引目标 | 快速定位符合条件的数据 | 快速找到距离最近的向量 |
| 排序依据 | 字段值、时间、业务规则 | 相似度分数、距离 |
| 结果特点 | 命中或不命中，通常确定性更强 | 返回最相似的一组结果，相关性需要评估 |
| 常见用途 | 交易系统、报表、后台管理 | RAG、语义搜索、推荐、去重、图文检索 |

举个新手容易理解的例子。

如果你问：

```text
员工张三今年还有几天年假？
```

这通常应该查传统数据库，因为你要的是精确、最新、可审计的数据。

如果你问：

```text
公司年假制度大概是怎么规定的？
```

这更适合 RAG，因为答案可能散落在制度文档、FAQ、员工手册中，需要先找相关段落，再组织语言回答。

## 5. 传统数据库 + 应用程序 vs 向量数据库 + LLM

很多人刚接触 RAG 时，会觉得“向量数据库 + LLM”很神秘。其实可以先用熟悉的应用架构做类比。

### 5.1 传统数据库 + 应用程序

一个普通业务系统通常是这样工作的：

```text
用户操作
  -> 应用程序解析意图
  -> 拼 SQL 或调用 ORM
  -> 数据库返回结构化数据
  -> 应用程序按规则渲染页面或返回 JSON
```

例如用户点击“查看订单详情”：

```text
应用程序知道要查 orders 表
应用程序用 order_id 做精确查询
数据库返回订单字段
应用程序展示订单详情页
```

这里的核心是：

- 应用程序掌握业务逻辑。
- 数据库负责可靠存储和精确查询。
- 返回结果通常是结构化的。

### 5.2 向量数据库 + LLM

RAG 应用通常是这样工作的：

```text
用户提问
  -> 应用程序把问题转成 embedding
  -> 向量数据库返回相似 chunk
  -> 应用程序组装 prompt
  -> LLM 基于上下文生成回答
```

例如用户问“新员工试用期请假有什么规则？”：

```text
应用程序把问题转成向量
向量数据库找到员工手册、请假制度、试用期政策中的相关段落
应用程序把这些段落放进 prompt
LLM 根据这些段落生成中文回答
```

这里的核心是：

- 应用程序负责流程编排、权限控制、prompt 构造和结果展示。
- 向量数据库负责语义检索。
- LLM 负责阅读上下文、归纳和生成自然语言答案。

### 5.3 对照表

| 传统系统 | RAG 系统 |
| --- | --- |
| 用户提交表单、点击按钮 | 用户用自然语言提问 |
| 应用程序拼 SQL | 应用程序生成 query embedding |
| 数据库按字段查数据 | 向量数据库按语义相似度找 chunk |
| 应用程序执行业务规则 | 应用程序组装 prompt、设置检索参数 |
| 页面模板或 API 返回结果 | LLM 生成自然语言答案 |
| 重点是数据准确和事务一致 | 重点是上下文相关、答案可追溯 |

可以把向量数据库理解成 RAG 应用里的“语义索引层”。它不替代应用程序，也不替代 LLM。

```text
传统数据库像“按编号和字段找资料的档案柜”。
向量数据库像“按意思找相近资料的语义地图”。
LLM 像“读完资料后帮你组织答案的助手”。
```

## 6. RAG 中每一步为什么重要

### 6.1 加载和清洗

RAG 的最终质量很大程度取决于原始数据质量。

如果文档重复、过时、结构混乱、权限不清晰，向量数据库也只会更快地把这些问题暴露出来。常见处理包括：

- 删除重复或明显过时的文档。
- 保留标题、路径、来源 URL、更新时间等元数据。
- 对 PDF、表格、网页等复杂格式做更可靠的解析。
- 对敏感内容做权限标记或脱敏。

### 6.2 切分 chunk

LLM 和 embedding 模型都不适合直接处理无限长文档，所以需要把文档切成 chunk。

chunk 太小，容易丢上下文：

```text
“审批人是直属经理。”
```

这句话单独看不知道在说什么审批。

chunk 太大，容易混入太多无关信息，检索命中后也会浪费上下文窗口。

本项目 `rag-demo t4` 专门对比了固定分块、语义分块、父子分块，并用 Recall@5 评估检索效果。

### 6.3 建立索引

建索引时，系统会对每个 chunk 生成 embedding，并把向量写入向量数据库。

传统数据库建索引通常是为了加速字段查询，例如 B-tree 索引加速 `WHERE created_at > ...`。

向量数据库建索引通常是为了加速最近邻搜索，例如 HNSW 或 IVF 类索引帮助快速找到相近向量。

本项目 `rag-demo t2` 用 Milvus 演示了 IVF_FLAT 和 HNSW 等索引参数对延迟、召回和资源使用的影响。

### 6.4 检索

用户提问时，系统会把问题也转成向量，然后查询最相似的 chunk。

常见参数包括：

| 参数 | 含义 |
| --- | --- |
| `top_k` | 返回最相似的前 K 个 chunk |
| metadata filter | 只查某类文档、某个租户、某个时间范围 |
| similarity threshold | 丢弃相似度太低的结果 |

Retriever、metadata filter、postprocessor 这些组件思路，在今天的 RAG 系统里很常见。

### 6.5 后处理和生成

检索结果不一定可以直接丢给 LLM。常见后处理包括：

- 过滤低相似度结果。
- 重新排序，让最相关内容更靠前。
- 根据权限或关键词排除不该进入 prompt 的内容。
- 补充前后相邻 chunk，让上下文更完整。

最后，应用程序把问题、检索到的上下文和回答要求组合成 prompt，让 LLM 生成最终答案。

## 7. 哪些内容需要更新理解

学习 RAG 时要注意区分“稳定概念”和“会变化的框架默认值”。

### 7.1 不要假设默认模型永远不变

旧材料常会写“LlamaIndex 默认使用 OpenAI 的某个模型”。这类说法容易过时。

更稳妥的理解是：

```text
RAG 框架需要一个 LLM 和一个 embedding 模型。
具体用哪个模型，应该由项目配置、成本、隐私、语言效果和部署方式决定。
```

本项目默认使用 Ollama 服务：

```text
Chat model: deepseek-v4-pro:cloud
Embedding model: qwen3-embedding:latest
```

在 `rag-demo t5` 中，还可以对比 BGE-M3 和 OpenAI `text-embedding-3-small` / `text-embedding-3-large`。

### 7.2 LlamaIndex 只是一个选择

不同 RAG 框架会用不同 API 表达同一条数据流。本项目的 T1 使用 LangChain、Chroma、LCEL 组织 RAG Hello World。

这不矛盾。不同框架的名字不同，但核心链路类似：

```text
读取文档 -> 切分 -> embedding -> 向量存储 -> 检索 -> prompt -> LLM
```

学概念时不要被框架 API 绑定。先理解数据流，再看具体代码。

### 7.3 RAG 不保证答案一定正确

RAG 能降低幻觉，但不能消灭幻觉。常见失败原因包括：

- 原始文档本身错了或过时。
- chunk 切分不合理。
- embedding 模型不适合当前语言或领域。
- top_k 太小，没有召回关键内容。
- top_k 太大，引入了噪声。
- prompt 没要求模型基于上下文回答。
- 用户问题需要精确实时数据，但系统只查了文档。

生产系统通常还需要引用来源、置信度、拒答策略、权限控制和评测集。

## 8. 什么时候用传统数据库，什么时候用向量数据库

可以用下面的判断规则：

| 问题类型 | 推荐方案 |
| --- | --- |
| 按 ID 查订单、库存、余额 | 传统数据库 |
| 复杂事务、写入一致性、报表统计 | 传统数据库 |
| 自然语言问制度、文档、知识库 | 向量数据库 + LLM |
| 找相似文章、相似图片、相似问题 | 向量数据库 |
| 既要语义搜索又要结构化过滤 | 向量数据库 + metadata filter，必要时结合传统数据库 |
| 需要精确实时业务数据并生成解释 | 传统数据库查事实，LLM 负责解释和表达 |

真实系统里经常是混合架构：

```text
PostgreSQL/MySQL 保存业务事实
对象存储保存原始文件
向量数据库保存 chunk embedding
应用程序负责权限、编排、评测和监控
LLM 负责生成自然语言答案
```


## 10. 小结

向量数据库不是“更高级的传统数据库”，而是面向相似度检索的专用基础设施。

传统数据库回答：

```text
哪些数据精确满足这些条件？
```

向量数据库回答：

```text
哪些内容和这个问题在语义上最接近？
```

LLM 则负责：

```text
读懂检索到的上下文，并生成用户能理解的答案。
```

理解 RAG 时，最重要的是把它看成一条工程链路，而不是单个模型能力：

```text
数据质量 + chunk 策略 + embedding 模型 + 向量数据库 + 检索参数 + prompt + LLM
```

其中任何一环做得不好，都会影响最终答案。
