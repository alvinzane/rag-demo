# LLM API 通信基础 & LLM 如何使用 RAG

这篇教程面向刚接触 LLM 应用开发的读者。读完后，你应该能回答四个问题：

1. 应用程序和 LLM API 到底在交换什么？
2. 一次普通 LLM 问答请求包含哪些关键字段？
3. RAG 为什么不是“把向量丢给 LLM”？
4. LLM 在 RAG 里具体负责哪一步？

本文会尽量用工程视角解释：LLM 是一个通过 API 调用的外部能力，RAG 是应用程序在调用 LLM 之前增加的一条检索链路。

## 1. 先理解 LLM API 是什么

LLM API 可以先理解成一个“远程文本生成服务”：

```text
你的应用程序
  -> 发送 HTTP JSON 请求
  -> LLM 服务读取 prompt、模型名和参数
  -> LLM 服务生成结果
  -> 返回 JSON 响应
```

这和调用普通业务 API 很像。

普通 API 可能是：

```text
GET /orders/123
返回订单 123 的结构化数据
```

LLM API 通常是：

```text
POST /chat 或 /responses
输入一段自然语言上下文
返回模型生成的自然语言结果
```

区别在于，普通业务 API 通常按确定的业务逻辑返回字段；LLM API 则根据输入上下文预测并生成文本。它不是数据库查询，也不是稳定的函数调用结果。

## 2. 一次 LLM API 请求里有什么

不同厂商的字段名会不同，但核心概念很接近。

| 字段 | 新手理解 | 例子 |
| --- | --- | --- |
| model | 用哪个模型回答 | `gpt-5`、`deepseek-v4-pro:cloud`、`qwen3` |
| messages/input | 发送给模型的内容 | 系统要求、用户问题、上下文 |
| temperature | 输出随机性 | `0` 更稳定，`0.8` 更发散 |
| max output tokens | 最多生成多长 | 限制回答长度和成本 |
| stream | 是否边生成边返回 | 聊天界面常用 |
| tools | 模型可调用的工具 | 搜索、数据库查询、函数调用 |

一个简化的请求可以这样理解：

```json
{
  "model": "some-chat-model",
  "messages": [
    {
      "role": "system",
      "content": "你是一个严谨的客服助手。"
    },
    {
      "role": "user",
      "content": "如何申请年假？"
    }
  ],
  "temperature": 0.2
}
```

模型返回的内容通常包含：

| 字段 | 含义 |
| --- | --- |
| id | 本次请求或响应 ID |
| output/message | 模型生成的内容 |
| usage | token 用量，常用于估算成本 |
| finish reason | 为什么停止生成，例如生成完成或达到长度限制 |
| tool calls | 如果启用工具调用，模型可能返回要调用哪个工具 |

## 3. role 是什么

很多聊天 API 会把输入拆成不同 role。新手可以这样理解：

| role | 作用 |
| --- | --- |
| system/developer | 给模型的高优先级规则，例如身份、边界、格式要求 |
| user | 用户真正提出的问题 |
| assistant | 模型之前的回答，用于多轮对话历史 |
| tool | 工具调用结果，例如搜索结果、数据库返回值 |

例如：

```text
system: 你只能根据给定资料回答，不确定就说不知道。
user: 新员工试用期能请假吗？
assistant: 可以，但需要参考公司制度。
user: 那需要谁审批？
```

第二个用户问题里的“那”依赖上一轮上下文。应用程序要么把历史对话一起发给模型，要么先把问题改写成完整问题：

```text
新员工试用期请假需要谁审批？
```

这也是很多对话式 RAG 系统会做“问题改写”的原因。

## 4. prompt 不只是用户问题

新手容易以为 prompt 就是用户输入的那句话。实际工程里，prompt 往往由应用程序拼出来：

```text
系统规则
+ 用户问题
+ 检索到的上下文
+ 输出格式要求
+ 安全边界
```

例如一个 RAG prompt 可以是：

```text
你是团队知识库问答助手。请只根据给定上下文回答问题。
如果上下文不足以回答，请直接说“不确定，当前索引中没有足够信息”。

上下文：
[source=员工手册.md chunk=12]
员工年假申请需要直属经理审批...

[source=试用期制度.md chunk=3]
试用期员工请假超过 2 天需要 HRBP 备案...

问题：
新员工试用期请假需要谁审批？
```

模型真正看到的是这整段内容，而不是只看到最后一句问题。

## 5. token 是什么，为什么它重要

LLM 不直接按“字数”处理文本，而是把文本切成 token。中文、英文、数字、标点都会被编码成 token。

token 重要有三个原因：

- 上下文窗口有限：一次请求能塞进去的内容有上限。
- 成本通常和 token 数有关：输入越长、输出越长，成本越高。
- 延迟也受影响：上下文越长，通常响应越慢。

这就是 RAG 需要“检索”的原因之一：不要把全部文档都塞给模型，而是先找出最相关的几段。

## 6. 普通 LLM 调用的局限

如果直接把用户问题发给 LLM，它只能依赖：

- 模型训练时学到的通用知识。
- 本次请求里你显式提供的上下文。
- 多轮对话里你保留并再次发送的历史。

它通常不知道：

- 你公司内部文档。
- 你的项目最新 README。
- 上周刚改的流程。
- 数据库里实时变化的订单、库存、工单状态。

所以直接调用 LLM API 很适合写作、总结、改写、翻译、通用问答；但要回答“基于私有知识库的问题”，就需要 RAG 或工具调用。

## 7. RAG 到底加了什么

RAG 的全称是 Retrieval-Augmented Generation，核心是：

```text
先检索相关资料，再让 LLM 基于资料生成回答。
```

它不是一个单独的模型，也不是一个神奇参数，而是一条应用链路：

```text
用户问题
  -> 应用程序生成 query embedding
  -> 向量数据库检索相关 chunk
  -> 应用程序组装 prompt
  -> 调用 LLM API
  -> LLM 生成回答
  -> 应用程序展示答案和来源
```

这里要特别注意：LLM 通常不会自己去向量数据库里查资料。大多数 RAG 应用中，是应用程序先完成检索，再把检索结果作为上下文发给 LLM。

## 8. LLM 在 RAG 里负责什么

LLM 在 RAG 中主要负责三类工作。

### 8.1 理解问题

LLM 可以帮助理解用户意图，尤其是在多轮对话里。

例如：

```text
用户第 1 轮：试用期员工怎么请假？
用户第 2 轮：超过两天呢？
```

第二轮问题单独看不完整。应用可以让 LLM 先改写为：

```text
试用期员工请假超过两天有什么要求？
```

然后再拿这个完整问题去检索。

### 8.2 阅读上下文

检索器返回的是若干 chunk，可能来自不同文件：

```text
chunk 1: 员工请假需要直属经理审批。
chunk 2: 试用期请假超过 2 天需要 HRBP 备案。
chunk 3: 年假余额可在 HR 系统中查询。
```

LLM 负责把这些片段读懂，判断哪些和问题有关，组织成自然语言答案。

### 8.3 生成回答

LLM 最后把上下文转换成用户能读懂的答案：

```text
试用期员工请假通常需要直属经理审批；如果请假超过 2 天，还需要 HRBP 备案。
当前资料没有说明是否需要部门负责人额外审批。
```

好的 RAG 回答应该尽量做到：

- 明确回答问题。
- 不把上下文没有的信息编出来。
- 在信息不足时直接说明。
- 给出来源，方便用户核对。

## 9. RAG 里哪些步骤不是 LLM 做的

为了避免误解，可以把职责拆开看。

| 步骤 | 主要负责人 |
| --- | --- |
| 读取文件、网页、数据库 | 应用程序、loader、connector |
| 清洗和切分文档 | 应用程序、splitter |
| 生成文档 embedding | embedding 模型 |
| 存储和索引向量 | 向量数据库 |
| 根据问题检索 chunk | retriever、向量数据库 |
| 组装 prompt | 应用程序 |
| 生成最终回答 | LLM |
| 展示答案、来源、上下文 | 应用程序 |

LLM 很重要，但它不是整套系统的全部。

## 10. LLM API 调用 vs RAG 调用

### 10.1 直接调用 LLM

```text
用户问题
  -> prompt
  -> LLM API
  -> 回答
```

适合：

- 改写文案。
- 总结已经给出的文本。
- 翻译。
- 写代码片段。
- 回答通用知识问题。

风险：

- 私有知识缺失。
- 最新信息缺失。
- 容易给出看似合理但无法溯源的答案。

### 10.2 使用 RAG 调用 LLM

```text
用户问题
  -> 检索知识库
  -> prompt + 检索上下文
  -> LLM API
  -> 带来源的回答
```

适合：

- 企业知识库问答。
- 项目文档助手。
- 客服知识库。
- 法规、制度、手册查询。
- 需要引用来源的问答系统。

风险：

- 检索错了，LLM 会基于错误上下文回答。
- chunk 切分不好，会丢失上下文。
- prompt 约束不清，模型可能补充未检索到的信息。
- 权限过滤不严，会把不该看的资料放进上下文。

## 11. LLM API 和 embedding API 的区别

RAG 通常会用到两类模型 API。

| API | 输入 | 输出 | 用途 |
| --- | --- | --- | --- |
| LLM/chat API | prompt、messages | 自然语言回答 | 生成答案 |
| embedding API | 文本 | 向量数组 | 语义检索 |

embedding API 不负责回答问题。它只负责把文本变成向量。

例如：

```text
输入：如何申请年假？
输出：[0.12, -0.08, 0.34, ...]
```

这个向量会被拿去和文档 chunk 的向量做相似度计算。

## 12. 当前 API 形态的几个更新点

LLM API 生态变化很快，学习时要抓住稳定概念，不要死记某个旧接口。

截至 2026-05，常见做法可以这样理解：

- OpenAI 新项目更推荐使用 Responses API；Chat Completions 仍被支持，但已不再是唯一入口。
- OpenAI Embeddings API 仍用于把文本转换为向量。
- Ollama 本地服务默认 API base URL 通常是 `http://localhost:11434/api`。
- Ollama 聊天接口是 `/api/chat`，embedding 新接口是 `/api/embed`；部分旧资料或兼容层可能还会提到 `/api/embeddings`。
- 很多框架会屏蔽底层 API 差异，但生产排错时仍需要理解 HTTP 请求、模型名、token、超时、流式响应和错误码。

所以看旧代码时，不要只问“这个接口还能不能用”，还要问：

```text
它是在做生成，还是在做 embedding？
它是一次性返回，还是 streaming？
它有没有传入历史对话？
它有没有传入检索上下文？
```

## 13. 在本项目里如何对应

本项目的 `rag-demo t1` 是一个最小 RAG 问答链路。核心流程可以对应到这些概念：

```text
Markdown 文档
  -> split_documents
  -> OllamaEmbeddings
  -> Chroma
  -> retriever.invoke(question)
  -> ChatPromptTemplate
  -> ChatOllama
  -> StrOutputParser
```

对应的工程职责是：

| 项目代码概念 | RAG 概念 |
| --- | --- |
| `load_markdown_documents_from_roots` | 加载原始文档 |
| `split_documents` | 文档切分成 chunk |
| `OllamaEmbeddings` | 生成 embedding |
| `Chroma` | 本地向量存储 |
| `store.as_retriever` | 检索相关 chunk |
| `format_context` | 把检索结果整理进 prompt |
| `ChatOllama` | 调用聊天模型生成答案 |

`ask_question` 里有一个很关键的提示词：

```text
请只根据给定上下文回答问题。
如果上下文不足以回答，请直接说“不确定，当前索引中没有足够信息”。
```

这句话的作用是给 LLM 设置边界：不要把没有检索到的内容当事实输出。

## 14. 一个完整请求的心智模型

假设用户问：

```text
团队发布流程是什么？
```

RAG 应用实际做的事是：

```text
1. 把问题转成 embedding。
2. 在向量数据库里找最相似的 4 个 chunk。
3. 把 chunk 格式化成“上下文”。
4. 拼出系统规则、上下文、用户问题。
5. 调用 LLM API。
6. 从响应 JSON 中取出回答文本。
7. 展示回答，同时展示 source 和 chunk_id。
```

LLM 只看到了第 4 步拼好的 prompt。它并不知道你的整个文件系统，也不会自动知道向量数据库里还有哪些内容。

## 15. 新手最容易混淆的点

### 15.1 RAG 不是微调

RAG 是“回答前查资料”。微调是“改变模型参数或行为倾向”。如果只是让模型回答你的文档内容，通常先考虑 RAG。

### 15.2 向量数据库不生成答案

向量数据库只返回相似内容。它不会写总结，也不会判断最终答案怎么表达。

### 15.3 LLM 不保证只说事实

即使有 RAG，LLM 仍可能误读上下文、过度推断或忽略约束。生产系统需要评测、日志、来源展示和人工兜底。

### 15.4 上下文越多不一定越好

塞太多 chunk 会增加成本和噪音。更好的做法是改进 chunk 策略、检索策略、metadata filter 和 rerank。

### 15.5 API 成功不代表答案正确

HTTP 200 只表示调用成功，不表示回答事实正确。LLM 应用要同时关注：

- 请求是否成功。
- 检索是否命中。
- 上下文是否足够。
- 回答是否忠于上下文。
- 来源是否可追溯。

## 16. 最小可运行练习

可以先用本项目跑一个本地 RAG 流程：

```bash
uv run rag-demo doctor
uv run rag-demo t1 index --docs ./confluence-export --persist ./.rag/index --reset
uv run rag-demo t1 ask "团队的发布流程是什么？" --persist ./.rag/index --show-context
```

观察三件事：

1. `--show-context` 展示的 chunk 是否真的和问题相关。
2. 回答是否严格来自这些 chunk。
3. sources 里的文件和 chunk_id 是否能帮助你追溯答案。

如果回答不好，先不要急着换大模型。优先检查：

- 文档是否被正确加载。
- chunk 是否切得合理。
- embedding 模型是否适合中文和当前领域。
- `top_k` 是否太小或太大。
- prompt 是否明确要求“不知道就说不知道”。

## 17. 总结

LLM API 通信的本质是：

```text
应用程序把 prompt 和参数发给模型服务，模型服务返回生成结果。
```

RAG 的本质是：

```text
应用程序先从知识库检索相关上下文，再把上下文交给 LLM 生成答案。
```

理解这两句话，就能避免很多常见误解：

- LLM API 不等于数据库查询。
- embedding API 不等于聊天 API。
- RAG 不是让 LLM 记住全部资料。
- LLM 在 RAG 里主要负责理解上下文和生成答案。
- 检索、权限、索引、来源追踪仍然是应用程序要负责的工程问题。

学习时先抓住这条主线：

```text
HTTP API -> prompt/messages -> embedding -> retriever -> context -> LLM -> answer with sources
```

后续再深入模型选择、流式响应、工具调用、rerank、评测和线上监控，会更容易建立完整的工程判断。
